## 6.1 Cron 调度

Hermes 的调度系统让 AI Agent 能够按照时间计划自动执行任务，充当你的"数字员工"。

### 6.1.1 /cron 命令基础

```bash
# 查看当前所有定时任务
/cron

# 创建定时任务（自然语言）
/cron add "每天早上9点给我发天气" "weather report"

# 列出所有任务
/cron list

# 删除任务
/cron delete <task-id>
```

### 6.1.2 自然语言任务创建

Hermes 支持用自然语言描述调度任务：

```bash
# 每日简报
/cron add "每个工作日早上8点发送今日新闻摘要" "news-digest"

/cron add "每周一早上10点总结上周工作" "weekly-summary"

/cron add "每月1号凌晨整理上月支出报表" "monthly-expense"

/cron add "每隔2小时检查服务器状态" "server-check"
```

### 6.1.3 调度语法解析

| 输入格式 | 解析结果 |
|----------|----------|
| `每天9点` | cron: `0 9 * * *` |
| `工作日早上8点` | cron: `0 8 * * 1-5` |
| `每周一10点` | cron: `0 10 * * 1` |
| `每月1号凌晨` | cron: `0 0 1 * *` |
| `每隔30分钟` | cron: `*/30 * * * *` |
| `每隔2小时` | cron: `0 */2 * * *` |

### 6.1.4 多平台投递

定时任务可以同时推送到多个平台：

```bash
/cron add "每天18点发送项目进度报告" "project-status" --platforms telegram,discord
```

配置示例：
```yaml
# config.yaml
cron:
  enabled: true
  default_platforms:
    - telegram
  max_concurrent: 5
  timeout_seconds: 300
```

### 6.1.5 自动化模板

Hermes 提供预置模板快速创建常见任务：

#### 日报模板
```bash
/cron add "工作日早上9点发送今日计划" --template daily-plan
```

#### PR 审查模板
```bash
/cron add "每天下午5点检查 GitHub PR 更新" --template pr-review
```

#### 备份模板
```bash
/cron add "每周日凌晨2点执行数据备份" --template backup
```

### 6.1.6 Cron 配置项详解

```yaml
# config.yaml
cron:
  enabled: true

  # 调度间隔
  check_interval: 60  # 每60秒检查一次

  # 任务超时
  task_timeout: 600  # 10分钟超时

  # 重试策略
  retry:
    max_attempts: 3
    backoff: exponential

  # 通知设置
  notification:
    on_success: false
    on_failure: true
    platforms:
      - telegram

  # 持久化
  storage: ~/.hermes/cron.db
```

## 6.2 委托与并行

### 6.2.1 delegate_task 委托工具

`delegate_task` 工具允许你将复杂任务委托给子 Agent 处理：

```python
# 工具签名
{
  "name": "delegate_task",
  "parameters": {
    "type": "object",
    "properties": {
      "task": {
        "type": "string",
        "description": "要委托的任务描述"
      },
      "context": {
        "type": "object",
        "description": "传递给子 Agent 的上下文"
      },
      "toolsets": {
        "type": "array",
        "description": "授权的工具集"
      }
    }
  }
}
```

使用示例：
```
用户: 请帮我分析这100个文件并生成报告

Hermes 使用 delegate_task:
- 主 Agent: 负责任务协调和最终汇总
- 子 Agent 1-5: 并行处理文件子集
- 每个子 Agent 返回: 局部分析结果
- 主 Agent: 汇总所有结果生成最终报告
```

### 6.2.2 execute_code 代码执行

`execute_code` 工具支持运行 Python、JavaScript 等代码：

```python
# 工具签名
{
  "name": "execute_code",
  "parameters": {
    "type": "object",
    "properties": {
      "language": {
        "type": "string",
        "enum": ["python", "javascript", "bash"]
      },
      "code": {
        "type": "string"
      },
      "timeout": {
        "type": "integer",
        "default": 60
      }
    }
  }
}
```

使用示例：
```
用户: 计算从1加到100的总和

Hermes 使用 execute_code(language="python", code="""
result = sum(range(1, 101))
print(f"总和: {result}")
""")
# 输出: 总和: 5050
```

### 6.2.3 /background 后台任务

将耗时任务放到后台执行：

```bash
# 启动后台任务
/backgrounde 生成一份详细的市场分析报告

# 查看后台任务
/background list

# 查看任务输出
/background output <task-id>

# 终止任务
/background kill <task-id>
```

后台任务特点：
- 非阻塞：立即返回，用户可继续其他操作
- 持久化：任务状态保存在数据库，重启不丢失
- 通知：任务完成时推送通知

### 6.2.4 并行工作流设计

#### 扇出模式（Fan-out）

```
主任务
   ├── 子任务 1 (并行)
   ├── 子任务 2 (并行)
   ├── 子任务 3 (并行)
   └── 汇总结果
```

实现：
```python
# 使用 delegate_task 实现并行
tasks = [
    {"task": "分析文件组A", "files": files_a},
    {"task": "分析文件组B", "files": files_b},
    {"task": "分析文件组C", "files": files_c}
]

results = await parallel_execute(
    [delegate_task(**t) for t in tasks]
)
final_report = aggregate(results)
```

#### 管道模式（Pipeline）

```
任务A → 任务B → 任务C → 最终结果
```

实现：
```python
# 顺序执行，每个任务依赖前一个输出
result_a = await task_a(data)
result_b = await task_b(result_a)
result_c = await task_c(result_b)
```

### 6.2.5 并发控制

```yaml
# config.yaml
concurrency:
  max_parallel_tasks: 5
  max_delegates: 3

  # 资源限制
  memory_per_task: "512MB"
  cpu_per_task: 1.0

  # 队列策略
  queue_strategy: priority  # priority, fifo, lifo
```

## 6.3 终端后端

### 6.3.1 环境抽象层

Hermes 通过抽象层支持多种执行环境：

```
┌─────────────────────────────────────────┐
│            Hermes Agent                  │
├─────────────────────────────────────────┤
│          Terminal Abstraction            │
├─────────┬─────────┬─────────┬────────────┤
│  Local  │ Docker  │ Cloud   │   SSH      │
│ Backend │ Sandbox │ Backend │  Remote    │
└─────────┴─────────┴─────────┴────────────┘
```

### 6.3.2 本地执行

默认使用本地 Shell 执行命令：

```bash
# 终端命令示例
$ ls -la
$ git status
$ python script.py
$ npm run build
```

配置：
```yaml
# config.yaml
terminal:
  backend: local

  local:
    shell: /bin/bash  # 或 /bin/zsh
    work_dir: ~
    env:
      PATH: /usr/local/bin:/usr/bin
```

### 6.3.3 Docker 沙箱

隔离环境执行命令，避免影响宿主机：

```yaml
# config.yaml
terminal:
  backend: docker

  docker:
    image: hermes-agent/sandbox:latest
    network: false        # 禁用网络
    volumes:              # 共享目录
      ~/projects: /workspace
    memory_limit: 2g
    cpu_limit: 1.0
    user: nonroot         # 以非 root 用户运行
```

常用场景：
- 执行未验证的代码
- 测试危险命令
- 隔离 Python 环境

### 6.3.4 云端执行

支持 Daytona、Modal、Singularity 等云端后端：

#### Daytona 配置
```yaml
terminal:
  backend: daytona

  daytona:
    enabled: true
    api_key: ${DAYTONA_API_KEY}
    server: https://app.daytona.io
    workspace_class: standard  # standard, performance
    prebuild_pool_size: 2
```

#### Modal 配置
```yaml
terminal:
  backend: modal

  modal:
    enabled: true
    api_token: ${MODAL_TOKEN}
    gpu: "T4"  # T4, A10G, A100
    memory: "16Gi"
```

#### Singularity 配置
```yaml
terminal:
  backend: singularity

  singularity:
    enabled: true
    endpoint: https://api.singularity.ai
    api_key: ${SINGULARITY_API_KEY}
```

### 6.3.5 SSH 远程执行

在远程服务器上执行命令：

```yaml
terminal:
  backend: ssh

  ssh:
    # 多个服务器配置
    hosts:
      - name: production
        host: 123.45.67.89
        port: 22
        user: deploy
        key_file: ~/.ssh/prod_key

      - name: staging
        host: 98.76.54.32
        port: 22
        user: deploy
        key_file: ~/.ssh/staging_key

    # 默认服务器
    default_host: production

    # 连接池
    pool_size: 5
    connection_timeout: 30
```

使用示例：
```
用户: 在生产服务器上重启 nginx

Hermes 使用:
hermes ssh --host production "sudo systemctl restart nginx"
```

### 6.3.6 后端选择策略

```yaml
# config.yaml
terminal:
  backend: auto  # 自动选择

  # 根据任务类型选择后端
  backend_routing:
    code_review: local
    untrusted_code: docker
    heavy_computation: daytona
    production_deploy: ssh
```

## 6.4 命令安全

### 6.4.1 危险命令检测

Hermes 内置危险命令检测机制：

| 类别 | 检测模式 | 示例 |
|------|----------|------|
| 递归删除 | `rm -rf /` 或 `rm -rf *` | 清除文件系统 |
| 磁盘操作 | `dd`, `fdisk`, `mkfs` | 格式化分区 |
| 权限修改 | `chmod 777` | 权限提升 |
| 网络操作 | `curl ... \| sh` | 远程代码注入 |

检测规则配置：
```yaml
# config.yaml
security:
  dangerous_detection:
    enabled: true
    severity: high

    patterns:
      - "rm\\s+-rf\\s+/"
      - "dd\\s+if="
      - "mkfs"
      - "curl.*\\|\\s*sh"
      - "wget.*\\|\\s*sh"
```

### 6.4.2 白名单/黑名单

```yaml
security:
  # 白名单：完全放行，无需审批
  whitelist:
    - echo
    - ls
    - pwd
    - cat
    - grep
    - git status
    - git diff

  # 黑名单：直接拒绝
  blacklist:
    - rm -rf /
    - dd if=
    - mkfs
    - fdisk
    - shred
    - "> /dev/sd"
```

### 6.4.3 /approve /deny 审批

命令执行前需要审批：

```
Hermes 检测到危险命令:
  命令: rm -rf ~/temp/*
  风险: 高 (递归删除)

需要审批:
  /approve <request-id>  # 批准执行
  /deny <request-id>     # 拒绝执行
  /deny --reason <reason> # 拒绝并说明原因
```

审批配置：
```yaml
# config.yaml
security:
  approval:
    enabled: true
    timeout_seconds: 300  # 5分钟超时

    # 审批人配置
    approvers:
      - user_id: admin
        level: full  # 可批准所有命令
      - user_id: operator
        level: limited  # 仅可批准低风险命令

    # 风险等级
    risk_levels:
      low:
        auto_approve: true
      medium:
        notify: true
      high:
        require_approval: true
      critical:
        require_manual_approval: true
```

### 6.4.4 YOLO 模式

绕过所有审批机制（谨慎使用）：

```bash
# 开启 YOLO 模式
/yolo

# 执行时临时启用
/hermes --yolo "rm -rf /tmp/test"
```

配置：
```yaml
security:
  yolo_mode:
    enabled: false  # 默认关闭
    require_pin: true  # 需要 PIN 码
    pin: ${YOLO_PIN}
    audit_log: true  # 记录所有操作
```

### 6.4.5 审批心跳机制

长时间运行的命令需要心跳确认：

```
任务开始:
  命令: ./long_running_script.sh
  预计时长: 30分钟
  心跳间隔: 60秒

等待心跳确认...
  ✓ 用户确认继续 (第1次)

任务进行中...
  ⚠ 心跳超时 (2分钟)
  任务已暂停
  /approve continue  # 恢复执行
```

配置：
```yaml
security:
  heartbeat:
    interval: 30  # 秒
    timeout: 120  # 2分钟无响应则暂停
    notification: true
```
